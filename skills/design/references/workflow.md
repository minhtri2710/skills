# Workflow

Use this flow for any non-trivial design task.

## Question Batch

Ask one compact batch instead of many back-and-forth messages.

Always cover:

1. design context
2. output scope
3. variation needs
4. editable or export requirements
5. task-specific unknowns

Recommended batch:

```md
Before I start, answer these in one pass:

1. Do you already have a design system, UI kit, codebase, or screenshots I should use?
2. What output do you want: prototype, deck, animation, infographic, or critique?
3. What level of fidelity do you want: placeholder, mid-fi, or full hi-fi?
4. Do you want multiple directions or one focused solution?
5. What must be included?
6. If this is branded work, do you have a logo, product images, UI screenshots, colors, or fonts?
7. If this is a deck, what final format matters: browser, PDF, or editable PPTX?
8. If this is motion work, what duration and destination matter?
```

## Junior Designer Loop

1. First pass:
   build structure, assumptions, and placeholders fast
2. First review:
   show it before polishing
3. Main build:
   replace placeholders with real content and stronger visuals
4. Variation pass:
   add alternate directions only if useful
5. Polish:
   refine typography, spacing, timing, and detail
6. Verify:
   check rendering, console health, interaction, and exports

## Variation Strategy

Variations should explore dimensions, not random cosmetic noise.

Good axes:

- typography
- density
- composition
- tone
- motion language
- color temperature
- material treatment

Good sets:

- conservative, balanced, bold
- editorial, system-driven, experimental
- sparse, medium, dense

Label each variation with what it explores.

## Branded Work

When a real brand is involved:

1. ask for assets once
2. search official sources for gaps
3. download and verify the assets
4. write `brand-spec.md` using `references/brand-spec-template.md`
5. build only after the spec exists

Mandatory assets:

- logo for any brand
- product images for physical products
- UI screenshots for digital products

## Delivery Summary

Keep it short:

- what was produced
- what remains placeholder
- what the user should review next

Avoid long implementation recaps.
